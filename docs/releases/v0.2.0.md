# r3f-interactive-flow v0.2.0

v0.2.0 is a stabilization release for the existing phase-based flow-control foundation. It focuses on making documented behavior, tests, and release-readiness notes clearer without introducing a major public API expansion.

## Highlights

- Stabilizes expectations for phase navigation, transition progress, input gating, cooldown, and lock behavior.
- Expands regression coverage across the core machine, React provider/hooks, the `useFlowFrame` bridge, and browser input hooks.
- Clarifies how DOM-facing progress differs from frame-driven progress consumed through `useFlowFrame`.
- Prepares release documentation required by the maintainer release checklist.

## Behavior stabilization

- Accepted navigation continues to publish the destination phase when a transition starts.
- Transition progress remains normalized to the `0..1` range and completes predictably.
- Valid navigation during an active transition, lock, cooldown, same-phase request, or phase boundary remains ignored rather than queued or retargeted.
- Invalid `goTo` targets continue to throw predictably without mutating state.
- First-phase `prev` and last-phase `next` remain non-looping no-ops.

## Testing improvements

- Core lifecycle regression coverage was added or expanded for initialization, `next`, `prev`, `goTo`, progress completion, lock behavior, cooldown behavior, boundary no-ops, and invalid targets.
- React provider and hook tests now cover mounted provider behavior, `useFlow`, `useFlowProgress`, snapshot updates, lock controls, and outside-provider errors.
- `useFlowFrame` tests cover frame updates, callback arguments, latest-callback handling, completion sync, and outside-provider errors.
- Wheel, touch, and keyboard input hook tests cover listener lifecycle, thresholds or key mappings, disabled state, prevention behavior, lock and transition gates, target elements, cleanup behavior, and import-time browser guards.
- Shared private input test utilities support the input hook tests. They are internal test utilities and are not part of the public API.

## Documentation improvements

- Transition lifecycle documentation now spells out accepted navigation, transition progress, completion, ignored navigation, cooldown, lock, boundary, and invalid-target behavior.
- Behavior documentation records the current v0.2.0 rules and test coverage expectations.
- README guidance clarifies that `useFlow` and `useFlowProgress` are DOM-facing snapshots, while `useFlowFrame` is the Canvas-bound frame bridge for visual updates.
- The v0.2.0 release readiness checklist documents the final maintainer validation pass before publishing.

## Scope and non-goals

- v0.2.0 does not introduce a major public API expansion.
- v0.2.0 does not add package exports, public source-phase or target-phase fields, or private test utilities to the public API.
- v0.2.0 does not add loop, queue, restart, interrupt, retarget, or `lockDuringTransition` behavior.
- v0.2.0 does not add particle, camera preset, shader, timeline, router, GSAP, or Framer Motion integration features.
- v0.2.0 does not add new runtime dependencies.

## Validation checklist

Before publishing, maintainers should complete the repository checklist in `docs/release-v0.2.0-checklist.md`, including package metadata review, public API confirmation, package output confirmation, example validation, and command results. Record any skipped checks with the reason they were skipped.

## Upgrade notes

- No major public API expansion is intended for v0.2.0.
- Existing users should review the transition lifecycle, input gating, cooldown, lock, and boundary behavior documentation.
- Existing users should review the distinction between DOM-facing progress from `useFlow`/`useFlowProgress` and frame-driven progress from `useFlowFrame`.
- These notes do not imply breaking changes beyond behavior already documented in this repository.
