# r3f-interactive-flow v0.4.0 Specification

## Status

This document defines the intended scope for `v0.4.0` before implementation work starts. It is a planning specification only: it does not change runtime behavior, package versioning, release automation, package exports, or published artifacts.

`v0.3.0` is the public API baseline. `v0.4.0` should harden that foundation through small, reviewable changes that can be split into follow-up issues and PRs.

## Theme

`v0.4.0 - Input, transition, and test hardening`

The release should stabilize existing behavior around phase navigation, transition lifecycle state, input hooks, cooldowns, locking, and test coverage. Adoption hardening is a supporting outcome: the package should become easier to trust and integrate because the existing behavior is clearer, better tested, and better documented.

## Goals

- Make `next`, `prev`, and `goTo` transition behavior predictable and test-backed.
- Clarify how manual locks, active transitions, and cooldowns interact.
- Improve wheel, touch, and keyboard input reliability without mixing input logic into scene logic.
- Expand core test coverage for accepted navigation, rejected navigation, timing, locking, and cooldown behavior.
- Keep README, examples, and release notes aligned with implemented behavior.
- Review public type exports so supported types are intentional and incidental internals remain private.
- Preserve the package's small public API and current architecture boundaries.
- Keep each implementation PR narrow enough to review independently.

## Non-goals

`v0.4.0` is not a feature-expansion release. Do not include:

- Visual effects collections.
- Particle APIs.
- Camera preset APIs.
- Shader effect APIs.
- Animation timeline systems.
- GSAP integration.
- Framer Motion integration.
- Router integration.
- Large demos, starters, or templates.
- New runtime dependencies unless a separate issue provides strong justification.
- Public API expansion unless a separate issue explicitly approves and documents it.
- Release automation changes unless explicitly requested.
- Package publishing, npm releases, GitHub releases, or Git tags.

## Package identity and architecture boundaries

Keep the package focused on phase-based interactive 3D website flow. Preserve the existing source layout:

```txt
packages/r3f-interactive-flow/
  src/
    core/
    react/
    r3f/
    input/
```

Boundary rules:

- `core/` stays React-independent.
- `react/` owns provider, context, and hooks.
- `r3f/` owns Canvas-bound frame bridge hooks.
- `input/` owns browser input handling.
- DOM input logic must not move into R3F scene logic.
- R3F hooks such as `useFrame` and `useThree` must not be called outside Canvas-bound components.
- Avoid React state for values that change every frame unless synchronizing a stable snapshot intentionally.
- Keep public exports small and intentional.
- Avoid broad rewrites and unrelated file changes.

## Current public API baseline after v0.3.0

The `v0.4.0` baseline is the public API exposed after `v0.3.0`:

Runtime exports:

- `FlowProvider`
- `useFlow`
- `useFlowProgress`
- `useFlowFrame`
- `useWheelInput`
- `useTouchInput`
- `useKeyboardInput`

Public type exports:

- `FlowFrameCallback`
- `FlowFrameState`
- `FlowInputTarget`
- `UseWheelInputOptions`
- `UseTouchInputOptions`
- `UseKeyboardInputOptions`
- `FlowTransitionBaseOptions`
- `FlowTransitionOptions`

Implementation work should treat this list as the compatibility target. Any proposed addition, removal, or rename must be handled as a separate public API review item with rationale, tests, documentation, and migration impact.

## Transition lifecycle hardening targets

Transition lifecycle work should verify and, where needed, harden:

- Accepted `next`, `prev`, and `goTo` navigation.
- Boundary no-ops at the first and last phase.
- Unknown phase handling for `goTo`.
- Transition progress advancing from `0` to `1`.
- Transition completion and stable final snapshots.
- Direction changes during accepted navigation.
- Direction reset to `"none"` after completion.
- Rejected navigation not mutating current phase, target phase, direction, progress, lock state, or cooldown state.
- Deterministic transition option resolution.
- `transition` and `transition.byPhase` behavior.

Expected outcome:

- Core navigation behavior is deterministic.
- Rejected navigation is safe and does not partially mutate state.
- Transition behavior remains independent from visual effects.
- Core transition tests can run without React or React Three Fiber.

## Locking and cooldown behavior targets

Locking and cooldown work should clarify and test:

- Manual `lock()` and `unlock()` behavior.
- Active transition guards.
- Cooldown start after accepted navigation.
- Ignored navigation not starting, resetting, or extending cooldown.
- Cooldowns longer than transition duration.
- Transition durations longer than cooldown.
- Input-level cooldown versus core-level cooldown.
- Default behavior for overlapping transitions.

Expected outcome:

- Locking rules are explicit and covered by tests.
- Cooldown behavior is predictable across accepted and rejected navigation.
- No new `lockDuringTransition` public API is introduced unless separately approved.

## Input hook reliability targets

Input hook work should focus on `useWheelInput`, `useTouchInput`, and `useKeyboardInput`.

Review and test:

- Listener attachment.
- Cleanup on unmount.
- `enabled` and disabled behavior.
- Target handling, including refs and direct targets.
- Threshold behavior.
- Axis handling.
- Ignore selectors.
- Keyboard repeat handling.
- Typing target guards.
- SSR and import safety.
- No browser API access at module import time.

Expected outcome:

- Input hooks translate browser input into flow navigation requests reliably.
- Input hooks do not own scene behavior.
- Input logic remains separate from R3F scene logic.
- Hook options stay focused and predictable.

## Core test coverage targets

Add or improve tests for:

- Initialization.
- Invalid phase lists.
- Invalid initial phase.
- `next`.
- `prev`.
- `goTo`.
- Boundary no-ops.
- Unknown phase errors.
- Transition progress and completion.
- Direction changes and reset.
- Manual lock and unlock.
- Active transition guard behavior.
- Cooldown start, advancement, and ignored requests.
- Invalid timing options.

Test guidance:

- Prefer tests before behavior changes when practical.
- Keep core tests React-independent.
- Use focused tests that make future refactors safer.
- Avoid broad snapshots that obscure behavior.

## Documentation and example improvement targets

Documentation and examples should be updated only when behavior is implemented or clarified. Keep the docs practical, direct, and implementation-focused.

Targets:

- Clarify expected transition behavior.
- Document input hook behavior that is covered by tests.
- Explain cooldown and locking behavior without overpromising advanced orchestration features.
- Add migration notes only if behavior changes.
- Keep examples small and aligned with the public API.
- Avoid marketing-heavy language and advanced animation promises.

## Public type export review targets

Review public type exports before release readiness:

- Confirm each exported type is intentionally supported.
- Avoid exporting incidental internal types.
- Document the purpose of supported public types where useful.
- Add type-level tests or export tests if an export is part of the public contract.
- Treat any public type removal or rename as a compatibility concern requiring separate review.

## Compatibility alias policy

Compatibility aliases should be conservative:

- Do not add aliases for new names unless a separate issue approves the naming and migration plan.
- Keep existing supported names stable through `v0.4.0` unless a breaking change is explicitly approved.
- If an alias is needed, document the canonical name, alias name, migration guidance, tests, and expected deprecation path.
- Do not use aliases to expand the conceptual API surface for this release.

## Testing and release-readiness expectations

Before `v0.4.0` release readiness, the following checks should pass when available:

- `pnpm format`
- `pnpm typecheck`
- `pnpm test`
- `pnpm lint`
- `pnpm build`

Release-readiness review should confirm:

- Runtime behavior changes are covered by tests.
- Documentation matches implemented behavior.
- Public exports are intentional.
- No runtime dependency was added without separate justification.
- Package version changes, publishing, npm releases, GitHub releases, and Git tags are not part of ordinary implementation PRs.

## Implementation guidance

Split implementation into small PRs that can be reviewed independently. Good follow-up issue or PR boundaries include:

1. Transition lifecycle tests and hardening for `next`, `prev`, and `goTo`.
2. Locking and cooldown tests with any necessary behavior fixes.
3. Wheel input reliability tests and fixes.
4. Touch input reliability tests and fixes.
5. Keyboard input reliability tests and fixes.
6. Public type export review and export tests.
7. README and example updates after behavior is implemented.
8. Final release-readiness checklist updates.

For each implementation PR:

- Keep the scope narrow.
- Start with tests when practical.
- Avoid unrelated refactors.
- Avoid new public APIs unless separately approved.
- Do not change package versions unless the PR is explicitly for release preparation.
- Report which checks were run and whether they passed.
