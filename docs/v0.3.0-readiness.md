## Behavior check

Expected behavior for the current v0.3.0 implementation:

- `next`, `prev`, and `goTo` still use the core phase machine.
- Boundary navigation remains a no-op.
- Same-phase `goTo()` remains a no-op.
- Invalid `goTo()` still throws.
- Navigation remains ignored while transitioning.
- Manual `lock()` / `unlock()` still control `isLocked`.
- Core cooldown starts only on accepted navigation.
- Core cooldown continues to elapse during transition updates.
- Ignored navigation does not start, reset, or extend core cooldown.
- Hook-local wheel/touch/keyboard cooldowns do not mutate core cooldown state.
- `transition.byPhase` uses source phase semantics.
- `transition` takes precedence over legacy timing props per field.
- `useFlowFrame` callback receives post-update frame state and delta.

## Breaking changes / migration

`useFlowFrame` callbacks now receive a frame state object as the first argument instead of the raw progress number.

Before:

```ts
useFlowFrame((progress, delta) => {
  // ...
});
```

After:

```ts
useFlowFrame(({ progress }, delta) => {
  // ...
});
```

Migration notes:

- `nextKeys` / `prevKeys` remain deprecated compatibility aliases.
- `keys.next` / `keys.prev` are the preferred keyboard configuration API.
- `transitionDurationMs`, `cooldownMs`, and `easing` remain legacy-compatible props.
- `transition` is the preferred timing API.
