# r3f-interactive-flow v0.3.0 Spec

## Status

Draft planning document. This is documentation-only and does not implement v0.3.0 behavior.

Target branch: `docs/add-v0.3.0-spec`.

## Background

`r3f-interactive-flow` is currently a small React Three Fiber utility for predictable phase-based interactive flow control. The README describes a narrow connector between user input, scene phases, transition progress, and R3F frame updates, not a visual effects collection or full animation framework.

v0.3.0 should improve practical phase control for real R3F websites while preserving that identity. The release should focus on phase management, transition progress, `next`, `prev`, `goTo`, browser input, DOM-to-Canvas usage, safer TypeScript types, small examples, and tests for core behavior.

## Goals

- Keep the public API centered on `FlowProvider`, `useFlow`, `useFlowProgress`, and `useFlowFrame`.
- Improve practical `next`, `prev`, and `goTo` behavior where timing, cooldown, and input locks interact.
- Add a safe design for phase-specific transition options.
- Improve wheel, touch, and keyboard input for real pages with controls, forms, links, modals, and ignored elements.
- Clarify the intended architecture: DOM React UI controls phase state; R3F scene components read phase/progress and own the visual behavior.
- Improve TypeScript type safety without casually expanding public API surface.
- Split implementation into small, testable PRs.
- Preserve browser and Next.js client safety without adding Next.js-specific integration.

## Non-Goals

v0.3.0 must not turn the package into an animation or effects framework. Do not add or design:

- particle morph APIs
- camera presets
- shader effects
- animation timelines
- router integration
- Next.js-specific router integration
- GSAP integration
- Framer Motion integration
- visual editors
- large demo templates
- full website templates
- heavy runtime dependencies
- new runtime dependencies

Current package exports:

- `FlowProvider`
- `useFlow`
- `useFlowProgress`
- `useFlowFrame`
- `useWheelInput`
- `useTouchInput`
- `useKeyboardInput`
- `UseWheelInputOptions`
- `UseTouchInputOptions`
- `UseKeyboardInputOptions`

Current `FlowProvider` state:

- The React entry uses `"use client"`.
- Props are `phases`, `initialPhase`, `transitionDurationMs`, `cooldownMs`, `easing`, and `children`.
- It creates one core machine with `useMemo`, stores a React snapshot, exposes controls through context, and syncs the snapshot after imperative navigation calls.
- There is no `transition` object prop and no per-phase transition option support.
- There is no `lockDuringTransition` prop. Current transition locking is effectively always-on because the core machine ignores new valid navigation while `isTransitioning` is true.

Current `useFlow` state:

- Reads `FlowContext` and returns `FlowControls<TPhase>`.
- Throws when used outside `FlowProvider`.
- Controls include snapshot fields plus `next`, `prev`, `goTo`, `lock`, and `unlock`.

Current `useFlowProgress` state:

- Reads `FlowContext` and returns only `context.controls.progress` as a number.
- Throws when used outside `FlowProvider`.

Current `useFlowFrame` state:

- Uses R3F `useFrame` and must run inside a Canvas-bound component.
- Current callback shape is `(progress: number, delta: number) => void`.
- It calls `machine.update(delta * 1000)` every frame, then calls the callback with the latest progress and R3F delta.
- It syncs the React snapshot when a transition changes from active to complete.
- Changing the callback to receive an object state would be breaking for current users.

Current core flow machine state:

- `FlowSnapshot<TPhase>` contains `phase`, `phaseIndex`, `progress`, `direction`, `isTransitioning`, and `isLocked`.
- `FlowControls<TPhase>` adds `next`, `prev`, `goTo`, `lock`, and `unlock`.
- `createFlowMachine` validates non-empty phases, duplicate phases, valid initial phase, positive finite transition duration, non-negative finite cooldown, and finite update deltas.
- `next`, `prev`, and valid `goTo` are no-ops when out of bounds, same phase, locked, transitioning, or cooling down.
- Invalid `goTo` throws even when locked, transitioning, or cooling down, without mutating state.
- `direction` is `"next"`, `"prev"`, or `"none"`.

Current transition duration behavior:

- Default duration is `1000ms`.
- Duration is configured through `transitionDurationMs`.
- Progress is computed from elapsed transition time and clamped to `0..1`.
- Easing defaults to linear and is applied to raw progress before completion.
- Progress becomes `1`, direction becomes `"none"`, and `isTransitioning` becomes `false` when raw progress reaches completion.

Current cooldown behavior:

- Default cooldown is `0ms`.
- Cooldown is configured through `cooldownMs`.
- Cooldown starts when navigation is accepted, not when transition completes.
- Cooldown time advances in `update(deltaMs)`, including while a transition is active.
- Ignored navigation does not start, reset, or extend cooldown.
- Negative update deltas do not reduce transition progress or cooldown time.

Current `lockDuringTransition` behavior:

- There is no public `lockDuringTransition` option.
- New navigation is currently always ignored during active transitions.
- `isLocked` represents manual `lock()` / `unlock()` only, not transition locking.

Current wheel input state:

- `useWheelInput` options are `target?: RefObject<HTMLElement | null>`, `threshold?: number`, `enabled?: boolean`, and `preventDefault?: boolean`.
- Default target is `window` when no target ref is provided.
- Default threshold is `40`.
- Default `preventDefault` is `true`.
- Only vertical `deltaY` is supported.
- Wheel down beyond threshold calls `next`; wheel up beyond threshold calls `prev`.
- It ignores input when the flow is locked or transitioning.
- It does not have `axis`, `cooldown`, `ignore`, direct `HTMLElement`, or explicit `Window` target typing.

Current touch input state:

- `useTouchInput` options are `target?: RefObject<HTMLElement | null>`, `threshold?: number`, `enabled?: boolean`, and `preventDefault?: boolean`.
- Default target is `window`.
- Default threshold is `50`.
- Default `preventDefault` is `true` on `touchmove`.
- Only vertical swipe is supported.
- Swipe up beyond threshold calls `next`; swipe down beyond threshold calls `prev`.
- It ignores input when the flow is locked or transitioning.
- It does not have `axis`, `cooldown`, `ignore`, direct `HTMLElement`, or explicit `Window` target typing.

Current keyboard input state:

- `useKeyboardInput` options are `target?: RefObject<HTMLElement | null>`, `enabled?: boolean`, `preventDefault?: boolean`, `nextKeys?: readonly string[]`, and `prevKeys?: readonly string[]`.
- Defaults: `ArrowDown`, `ArrowRight`, `PageDown`, and Space for next; `ArrowUp`, `ArrowLeft`, and `PageUp` for prev.
- Repeated keydown events are ignored.
- Keyboard events from `input`, `textarea`, `select`, or contenteditable elements are always ignored.
- Default `preventDefault` is `true` for mapped keys.
- It ignores input when the flow is locked or transitioning.
- It does not have `keys: { next, prev }`, `cooldown`, or configurable `ignoreWhenTyping`.

Current TypeScript state:

- Phase names are generic through `TPhase extends string`.
- The package exports input option types but not `FlowSnapshot`, `FlowControls`, or transition option types from the root entry.
- Input `target` types currently only accept React refs.
- `useFlowFrame` does not expose a frame state type because its callback receives `progress` and `delta` as positional arguments.

Current test coverage:

- Core tests cover initialization, invalid phases, `next`, `prev`, `goTo`, boundaries, locks, transition progress, easing, cooldown timing, ignored navigation, invalid `goTo`, and validation errors.
- React provider tests cover context behavior, controls, progress, transitions, lock/unlock, cooldown, memoization/snapshot behavior, and hook errors.
- R3F tests mock `useFrame` and cover frame registration, progress callback behavior, transition completion syncing, callback updates, and provider errors.
- Wheel tests cover listener attachment, threshold, next/prev mapping, disabled state, default/prevented events, lock/transition guards, cleanup, browser API import safety, and ref targets.
- Touch tests cover listener attachment, ref targets, default/prevented touchmove, threshold, swipe up/down, disabled state, lock/transition guards, cleanup, touch cancel, and browser API import safety.
- Keyboard tests cover listener attachment, default keys, unmapped/repeat ignored events, typing targets, disabled state, preventDefault, custom key arrays, lock/transition guards, cleanup, browser API import safety, and ref targets.

Current examples:

- `examples/vite-basic` demonstrates `FlowProvider`, input hooks, DOM controls, and `useFlowFrame((progress) => ...)` with a small Canvas scene.

## Proposed Scope

v0.3.0 should include:

1. Safer scoped wheel input options.
2. Practical touch input options.
3. Keyboard input guard improvements.
4. A stable `useFlowFrame` frame state payload, with an explicit migration note if breaking.
5. Phase-specific transition options through a clear transition configuration object.
6. A small DOM UI to Canvas usage example in docs/README/example code.
7. Tests for all behavior changes.

v0.3.0 should not include runtime dependency changes, public export churn beyond justified types/options, feature implementation unrelated to phase/input/frame control, or large example rewrites.

## Proposed API Direction

Any proposed public API change must include a reason, expected behavior, type impact, README impact, migration impact, and whether it is breaking.

### Phase-specific transition options

Proposed public direction:

```tsx
<FlowProvider
  phases={["intro", "skills", "projects", "contact"] as const}
  transition={{
    duration: 1000,
    cooldown: 500,
    lockDuringTransition: true,
    byPhase: {
      intro: {
        duration: 1600,
        cooldown: 800
      },
      skills: {
        duration: 800
      },
      contact: {
        cooldown: 1200
      }
    }
  }}
>
  <App />
</FlowProvider>
```

Suggested type direction:

```ts
type FlowTransitionBaseOptions = {
  duration?: number;
  cooldown?: number;
  lockDuringTransition?: boolean;
  easing?: FlowEasing;
};

type FlowTransitionOptions<TPhase extends string> = FlowTransitionBaseOptions & {
  byPhase?: Partial<Record<TPhase, FlowTransitionBaseOptions>>;
};
```

Expected behavior:

- A transition starting from `intro` uses `transition.byPhase.intro` overrides.
- A transition starting from `skills` uses `transition.byPhase.skills` overrides.
- If no phase override exists, global transition settings are used.
- Override resolution is based on the source phase before navigation mutates `phaseIndex`.
- `duration`, `cooldown`, `lockDuringTransition`, and `easing` fall back independently from phase override to global transition settings to defaults.
- Invalid override values should throw with specific messages during machine/provider setup or accepted navigation resolution, not silently clamp.

Reason:

- Real sites often need a slower introduction, faster interior phases, or longer cooldown for contact/footer areas.

Type impact:

- Add transition option types, ideally exported if users need to annotate reusable config.
- Consider retaining `transitionDurationMs`, `cooldownMs`, and `easing` during v0.3.0 as compatibility aliases, but document `transition` as the preferred direction.

README impact:

- Replace or supplement `transitionDurationMs` / `cooldownMs` docs with `transition` examples.
- Document source-phase semantics explicitly.

Migration impact:

- If compatibility aliases remain, existing users can migrate gradually from `transitionDurationMs={1000}` and `cooldownMs={500}` to `transition={{ duration: 1000, cooldown: 500 }}`.
- If aliases are removed, this is breaking and must be called out prominently. Recommendation: do not remove aliases in the same PR that introduces `transition` unless the maintainer intentionally accepts a v0.x breaking cleanup.

Breaking:

- Adding `transition` is not breaking if existing props remain.
- Removing or renaming existing timing props is breaking.

### Scoped wheel input

Proposed public direction:

```tsx
function FlowInputLayer() {
  const containerRef = useRef<HTMLDivElement>(null);

  useWheelInput({
    target: containerRef,
    threshold: 48,
    cooldown: 700,
    axis: "y",
    enabled: true,
    ignore: ["input", "textarea", "select", "button", "[data-flow-ignore]"]
  });

  return (
    <div ref={containerRef}>
      <CanvasScene />
      <Overlay />
    </div>
  );
}
```

Suggested type direction:

```ts
type FlowInputTarget = React.RefObject<HTMLElement | null> | HTMLElement | Window;

type UseWheelInputOptions = {
  target?: FlowInputTarget;
  enabled?: boolean;
  threshold?: number;
  cooldown?: number;
  axis?: "x" | "y";
  preventDefault?: boolean;
  ignore?: string[];
};
```

Expected behavior:

- Default target remains `window`.
- `target` accepts a ref, direct element, or `window`.
- `axis: "y"` uses `deltaY`; `axis: "x"` uses `deltaX`.
- Positive selected-axis delta beyond threshold calls `next`; negative selected-axis delta beyond threshold calls `prev`.
- Events at or inside threshold are ignored.
- `ignore` selectors prevent navigation and should run before `preventDefault` where possible, so ignored form controls keep normal browser behavior.
- Default ignore list should include at least interactive controls and `[data-flow-ignore]`, or the README should recommend that list if defaulting is considered too risky.
- Hook-local `cooldown` throttles accepted wheel navigation events even if the core machine cooldown is `0`.

Reason:

- Current window-level vertical wheel behavior is too broad for real pages with forms, buttons, links, modals, and scrollable areas.

Type impact:

- Extends `UseWheelInputOptions` and adds `FlowInputTarget` if exported.

README impact:

- Update input hook options and show scoped container usage.

Migration impact:

- Existing options continue to work if defaults are preserved.

Breaking:

- Not breaking if existing defaults remain, but adding a default ignore list could change behavior for users who relied on wheel over buttons/forms. If default ignore is added, document it as a behavior change.

### Touch input

Proposed public direction:

```tsx
useTouchInput({
  target: containerRef,
  axis: "y",
  threshold: 64,
  cooldown: 700,
  enabled: true,
  preventDefault: true,
  ignore: ["button", "a", "[data-flow-ignore]"]
});
```

Expected behavior:

- Default target remains `window`.
- `axis: "y"`: swipe up calls `next`; swipe down calls `prev`.
- `axis: "x"`: swipe left calls `next`; swipe right calls `prev`.
- Events at or inside threshold are ignored.
- Ignore selectors prevent navigation and should preserve normal interaction behavior.
- Hook-local cooldown throttles accepted swipe navigation.
- Default `preventDefault` should become conservative. Recommendation: default to `false` for new v0.3.0 behavior unless maintainers decide backward compatibility with the current `true` default is more important.

Reason:

- Current touch support only handles vertical movement and defaults to preventing browser scroll, which can be surprising for real websites.

Type impact:

- Extend `UseTouchInputOptions` with `axis`, `cooldown`, and `ignore`.
- Extend target type as with wheel.

README impact:

- Document swipe direction table and default `preventDefault` behavior.

Migration impact:

- If default `preventDefault` changes from `true` to `false`, users who depended on scroll blocking must set `preventDefault: true` explicitly.

Breaking:

- Changing `preventDefault` default is behavior-breaking. If done, include a migration note.

### Keyboard input

Proposed public direction:

```tsx
useKeyboardInput({
  keys: {
    next: ["ArrowDown", "PageDown", "Space"],
    prev: ["ArrowUp", "PageUp"]
  },
  enabled: true,
  ignoreWhenTyping: true
});
```

Suggested type direction:

```ts
type UseKeyboardInputOptions = {
  enabled?: boolean;
  keys?: {
    next?: string[];
    prev?: string[];
  };
  cooldown?: number;
  ignoreWhenTyping?: boolean;
};
```

Recommended compatibility type:

```ts
type UseKeyboardInputOptions = {
  target?: FlowInputTarget;
  enabled?: boolean;
  preventDefault?: boolean;
  keys?: {
    next?: readonly string[];
    prev?: readonly string[];
  };
  cooldown?: number;
  ignoreWhenTyping?: boolean;
  /** @deprecated Use keys.next. */
  nextKeys?: readonly string[];
  /** @deprecated Use keys.prev. */
  prevKeys?: readonly string[];
};
```

Expected behavior:

- `ignoreWhenTyping: true` ignores focus inside `input`, `textarea`, `select`, and `[contenteditable]`.
- Because current behavior already always ignores typing targets, recommended default is `true`.
- `ignoreWhenTyping: false` allows mapped keys from typing targets only when intentionally opted in.
- Keep repeated keydown ignored.
- Hook-local cooldown throttles accepted keyboard navigation.

Reason:

- The current keyboard guard is sensible, but not configurable and uses `nextKeys` / `prevKeys` rather than a grouped `keys` object.

Type impact:

- Adds `keys`, `cooldown`, and `ignoreWhenTyping`.
- Retaining `nextKeys` / `prevKeys` as deprecated aliases avoids immediate breakage.

README impact:

- Update examples to `keys.next` / `keys.prev` if adopted.

Migration impact:

- Existing `nextKeys` / `prevKeys` can keep working during v0.3.0 with deprecation notes.

Breaking:

- Removing `nextKeys` / `prevKeys` is breaking. Recommendation: do not remove them in v0.3.0 unless maintainers explicitly choose a breaking cleanup.

### useFlowFrame frame state

Proposed public direction:

```tsx
useFlowFrame(({ phase, phaseIndex, progress, direction, isTransitioning }) => {
  if (phase === "skills") {
    meshRef.current.rotation.y += 0.01;
  }
});
```

Suggested type direction:

```ts
type FlowFrameState<TPhase extends string> = {
  phase: TPhase;
  phaseIndex: number;
  progress: number;
  direction: "next" | "prev" | "none";
  isTransitioning: boolean;
};
```

Important current-state constraint:

- Current `useFlowFrame` callback is `(progress: number, delta: number) => void`.
- Changing the first argument from `number` to `FlowFrameState` is breaking.

Recommended design:

```ts
type FlowFrameCallback<TPhase extends string> = (
  state: FlowFrameState<TPhase>,
  delta: number
) => void;
```

Expected behavior:

- The callback receives a freshly read snapshot after `machine.update` for the frame.
- `delta` remains available as the second argument to preserve the useful R3F frame delta.
- `FlowFrameState` should be stable in shape, not necessarily referentially stable per frame.
- The state should not expose mutation methods; `useFlowFrame` remains a read/update bridge, not a control hook.

Reason:

- Real R3F scenes usually need `phase`, `phaseIndex`, `progress`, `direction`, and transition status together, not only numeric progress.

README impact:

- Update R3F examples from `useFlowFrame((progress, delta) => ...)` to `useFlowFrame(({ phase, progress, direction }, delta) => ...)`.

Migration impact:

- Before: `useFlowFrame((progress, delta) => { ... })`.
- After: `useFlowFrame(({ progress }, delta) => { ... })`.
- If backward compatibility is desired, support both callback forms temporarily, but overloads add complexity and may hide migration mistakes.

Breaking:

- Breaking if callback argument changes. Because the package is still `0.x`, this may be acceptable if clearly documented.

### DOM UI to Canvas usage pattern

Recommended architecture:

- DOM React UI controls phase state.
- R3F scene reads phase/progress through hooks.
- The library does not provide visual effects.
- The user owns animation and scene logic.

Conceptual DOM control:

```tsx
function FlowNav() {
  const { phase, phaseIndex, next, prev, goTo, isTransitioning } = useFlow();

  return (
    <nav>
      <p>
        Current phase: {phase} / {phaseIndex}
      </p>
      <button onClick={prev} disabled={isTransitioning}>
        Prev
      </button>
      <button onClick={next} disabled={isTransitioning}>
        Next
      </button>
      <button onClick={() => goTo("projects")} disabled={isTransitioning}>
        Go to Projects
      </button>
    </nav>
  );
}
```

Conceptual R3F usage:

```tsx
function FlowObject() {
  const meshRef = useRef<THREE.Mesh>(null);

  useFlowFrame(({ phase, progress, direction }) => {
    if (!meshRef.current) return;

    if (phase === "intro") {
      meshRef.current.position.z = -4 + progress * 4;
    }

    if (phase === "skills") {
      meshRef.current.rotation.y += 0.02;
    }

    if (phase === "projects") {
      const sign = direction === "prev" ? -1 : 1;
      meshRef.current.position.x = progress * sign * 2;
    }
  });

  return (
    <mesh ref={meshRef}>
      <boxGeometry />
      <meshStandardMaterial />
    </mesh>
  );
}
```

## Behavior Details

### Transition resolution

- Resolve transition options at the moment a navigation command is accepted.
- Use the source phase for `byPhase` lookup.
- Resolution order per field:
  1. `transition.byPhase[sourcePhase]?.field`
  2. `transition.field`
  3. existing default
- Proposed field defaults:
  - `duration`: `1000`
  - `cooldown`: `0`
  - `lockDuringTransition`: `true`, to match current always-ignore-during-transition behavior
  - `easing`: linear
- `duration` must be finite and greater than `0`.
- `cooldown` must be finite and greater than or equal to `0`.
- `easing` must be callable.
- If both legacy props and `transition` are provided, pick one explicit precedence and warn in documentation. Recommendation: `transition` wins, legacy props are compatibility aliases.

### Cooldown behavior

- Core cooldown starts only when navigation is accepted.
- Ignored navigation must not start, reset, or extend core cooldown.
- Cooldown continues to elapse during transitions.
- With `transition.byPhase`, the cooldown value should come from the accepted transition's source phase resolution.
- Hook-local cooldowns for wheel/touch/keyboard should throttle only that input hook and should not mutate or expose core cooldown state.
- Hook-local cooldown should start only after the hook attempts an accepted navigation command. Testing may need a command result; see open questions.

### Input lock behavior

- Manual `lock()` / `unlock()` should continue to control `isLocked`.
- Input hooks should continue to ignore navigation when `flow.isLocked` is true.
- `lockDuringTransition` should control whether new valid navigation is ignored while transitioning.
- To preserve current behavior, default `lockDuringTransition` should be `true`.
- If `lockDuringTransition: false` is supported, define whether interrupting transitions is allowed. Recommendation for v0.3.0: do not implement transition interruption; instead treat this option as future-facing only if interruption semantics are not ready. If exposed, it must be fully tested.

### Input ignore behavior

- Ignore checks should support selector strings and nearest ancestor matching via `Element.closest`.
- Ignored events should not call `next`, `prev`, or `goTo`.
- Ignored events should avoid `preventDefault` unless a specific option intentionally requests prevention even on ignored targets.
- Recommended ignore selectors for docs:
  - `input`
  - `textarea`
  - `select`
  - `button`
  - `a`
  - `[contenteditable]`
  - `[data-flow-ignore]`
  - optionally `[data-flow-scroll]` for scrollable areas if documented
- Ignore checks must be safe when `event.target` is not an `Element`.

### Browser API safety

- Public React/R3F entry files may need `"use client"` when appropriate.
- Do not access `window`, `document`, `HTMLElement`, `Element`, `TouchEvent`, `WheelEvent`, `KeyboardEvent`, or browser event APIs at module import time.
- Use browser APIs only inside effects, event handlers, or guarded client-side code.
- Keep import-time tests for browser API safety.
- Do not add Next.js as a dependency.
- Do not add Next.js-specific router integration.

## Type Design

Recommended types:

```ts
type FlowDirection = "next" | "prev" | "none";

type FlowEasing = (progress: number) => number;

type FlowTransitionBaseOptions = {
  duration?: number;
  cooldown?: number;
  lockDuringTransition?: boolean;
  easing?: FlowEasing;
};

type FlowTransitionOptions<TPhase extends string> = FlowTransitionBaseOptions & {
  byPhase?: Partial<Record<TPhase, FlowTransitionBaseOptions>>;
};

type FlowFrameState<TPhase extends string> = {
  phase: TPhase;
  phaseIndex: number;
  progress: number;
  direction: FlowDirection;
  isTransitioning: boolean;
};

type FlowInputTarget = React.RefObject<HTMLElement | null> | HTMLElement | Window;
```

Type recommendations:

- Keep `TPhase extends string` and preserve literal phase inference from `as const` phase tuples.
- Consider exporting `FlowDirection`, `FlowTransitionOptions`, `FlowFrameState`, and `FlowInputTarget` if users need to type reusable config/callbacks. Each export should be intentional and covered by `index.test.ts` or type tests.
- Avoid exposing the core machine as a new public API unless already intentionally public. Current root entry does not export `createFlowMachine`.
- Keep input option changes additive where possible.
- Prefer `readonly string[]` in option types where arrays are not mutated.

## Breaking Changes and Migration Notes

Potential breaking changes:

1. `useFlowFrame` callback payload changing from `(progress, delta)` to `(state, delta)`.
   - Migration: `useFlowFrame((progress, delta) => ...)` becomes `useFlowFrame(({ progress }, delta) => ...)`.
   - Reason: scenes need phase, index, direction, and transition state.
   - Breaking: yes.
2. Touch `preventDefault` default changing from `true` to `false`.
   - Migration: set `preventDefault: true` explicitly to preserve scroll-blocking behavior.
   - Reason: safer default for real pages.
   - Breaking: behavior-breaking.
3. Keyboard `nextKeys` / `prevKeys` replaced by `keys.next` / `keys.prev`.
   - Migration: `nextKeys: [...]` becomes `keys: { next: [...] }`.
   - Recommendation: keep deprecated aliases in v0.3.0 to avoid immediate breakage.
   - Breaking: only if aliases are removed.
4. `transitionDurationMs` / `cooldownMs` replaced by `transition.duration` / `transition.cooldown`.
   - Migration: `transitionDurationMs={1000}` and `cooldownMs={500}` become `transition={{ duration: 1000, cooldown: 500 }}`.
   - Recommendation: keep compatibility aliases in v0.3.0.
   - Breaking: only if old props are removed.
5. Default input ignore behavior.
   - Migration: if defaults ignore controls/links/forms, users can customize `ignore` if they need different behavior.
   - Breaking: potentially behavior-breaking if defaults change navigation behavior over interactive elements.

## Implementation Plan

1. Add shared input target and ignore utilities internally.
2. Improve wheel options with target resolution, axis, ignore selectors, and hook cooldown.
3. Improve touch options with axis, ignore selectors, hook cooldown, and documented `preventDefault` behavior.
4. Improve keyboard options with grouped keys, configurable typing guard, and hook cooldown while retaining aliases if desired.
5. Stabilize `useFlowFrame` around a typed state payload and update tests/docs.
6. Add transition option resolution in core machine and provider without changing public exports prematurely.
7. Update README/docs with DOM UI to Canvas usage and migration notes.
8. Keep all implementation PRs small and test-first where practical.

## Risks

- `useFlowFrame` state payload is useful but breaking.
- `lockDuringTransition: false` may imply interruptible transitions, which are not currently designed. Shipping the option without clear behavior would be risky.
- Hook-local cooldown may be hard to make exact because current controls return `void`, so hooks cannot know whether navigation was accepted. A command result type may be needed but would affect public API.
- Changing touch `preventDefault` default improves page safety but breaks users relying on current scroll prevention.
- Default ignore selectors improve practical safety but can change existing behavior.
- Supporting both legacy and new option names can create precedence complexity.
- Exporting more types improves usability but expands public API maintenance.

## Open Questions

1. Should `useFlowFrame` make the breaking jump to `(state, delta)` in v0.3.0, or temporarily support both callback shapes?
2. Should `transitionDurationMs` / `cooldownMs` remain as aliases through v0.3.0?
3. Should touch `preventDefault` default change to `false`, or remain `true` for compatibility?
4. Should ignore selectors have a built-in default list, or should examples recommend a list while defaulting to no ignore list?
5. Should command methods return a boolean or result object to indicate accepted vs ignored navigation? This would help hook-local cooldown correctness but changes the API.
6. Should `lockDuringTransition: false` be deferred until interrupt/restart semantics are explicitly designed?
7. Which new types should be exported from the package root?
8. Should keyboard `Space` use the actual `KeyboardEvent.key` value `" "` or support `"Space"` as a convenience alias in docs/API?
