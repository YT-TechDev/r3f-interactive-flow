# r3f-interactive-flow v0.3.0 Specification

## Status

This document is the design baseline for `r3f-interactive-flow` v0.3.0. It is intentionally documentation-only: it describes the expected behavior and responsibility boundaries future implementation PRs should follow, but it does not introduce new runtime behavior by itself.

## Project direction

`r3f-interactive-flow` is a small React Three Fiber utility library for phase-based interactive 3D websites. Its purpose is to connect user input, scene phases, transition progress, React state, and React Three Fiber frame updates through a predictable control layer.

The library should stay focused on:

- user input
- scene phases
- transition progress
- React state
- React Three Fiber frame-based updates

The library must not become:

- a visual effects collection
- a `@react-three/drei` replacement
- a particle library
- a camera preset library
- a shader effect library
- a portfolio template
- a full animation framework
- a GSAP or Framer Motion wrapper

## v0.3.0 goals

v0.3.0 should clarify and strengthen the foundation for interactive phase-based R3F experiences.

Goals:

- Define predictable transition lifecycle semantics for `next`, `prev`, and `goTo`.
- Keep input handling explicit, guarded, and separate from scene rendering logic.
- Clarify how transition locking and cooldowns interact.
- Preserve a small public API centered on `FlowProvider`, `useFlow`, `useFlowProgress`, and `useFlowFrame`.
- Keep React responsibilities separate from R3F frame responsibilities.
- Keep frame-driven values out of React state unless a React snapshot update is intentionally needed.
- Maintain strong TypeScript inference for phase names and flow state.
- Provide implementation guidance so future PRs can be small, reviewable, and testable.

## Scope

v0.3.0 is in scope for behavior and documentation around:

- phase registration and current phase state
- `next`, `prev`, and `goTo` navigation
- transition progress from `0` to `1`
- transition direction
- transition lifecycle completion
- manual flow locking
- input-triggered navigation guards
- cooldown behavior after accepted navigation
- React provider and hook responsibilities
- React Three Fiber bridge responsibilities
- TypeScript public and internal type expectations
- tests for core, React, R3F bridge, and input behavior

Documentation for v0.3.0 may describe future implementation intent, but implementation should happen in separate PRs unless a future issue explicitly scopes code changes.

## Non-goals

v0.3.0 must not add broad animation or rendering features. The following are out of scope:

- built-in visual effects
- particle systems
- camera preset APIs
- shader effect APIs
- material or postprocessing collections
- timeline authoring tools
- GSAP wrappers
- Framer Motion wrappers
- router integrations
- Next.js-specific routing behavior
- portfolio or landing-page templates
- large demo rewrites unrelated to the core flow model
- new runtime dependencies unless separately justified

## Architecture boundaries

The expected source layout is:

```txt
packages/r3f-interactive-flow/
  src/
    core/
    react/
    r3f/
    input/
```

Responsibilities must remain separated:

- `core/` contains React-independent TypeScript logic.
- `react/` contains React Context, Provider, and hooks.
- `r3f/` contains React Three Fiber bridge hooks.
- `input/` contains browser input handling.

Rules:

- Do not mix DOM input logic into R3F scene logic.
- Do not call R3F hooks such as `useFrame` or `useThree` outside Canvas-bound components.
- Do not use React state for values that change every frame unless a React snapshot is deliberately being synchronized.
- Use refs or internal mutable state for frame-driven values when appropriate.
- Keep public API additions rare, small, and predictable.

## Transition lifecycle behavior

A transition represents an accepted navigation from one phase to another.

Expected lifecycle:

1. The flow starts idle at an initial phase.
2. A navigation request is made through `next`, `prev`, or `goTo`.
3. The request is validated against bounds, known phases, lock state, transition state, and cooldown state.
4. If accepted, the transition records its source phase, target phase, direction, duration, easing, and cooldown settings.
5. The current phase updates to the target phase according to the core flow model.
6. `progress` starts at `0` for the accepted transition.
7. Frame updates advance elapsed transition time.
8. `progress` is clamped to the inclusive range `0..1`.
9. When the transition reaches completion, `progress` becomes `1`, `isTransitioning` becomes `false`, and `direction` returns to `"none"`.
10. React state is synchronized at stable boundaries, especially after navigation calls and when a frame-driven transition completes.

A rejected navigation request must not partially mutate transition state.

## Phase transition rules

Navigation rules:

- `next` requests the next phase by index.
- `prev` requests the previous phase by index.
- `goTo` requests a specific named phase.
- `next` at the final phase is a no-op.
- `prev` at the first phase is a no-op.
- `goTo` for the current phase is a no-op.
- `goTo` for an unknown phase should fail clearly rather than silently selecting a fallback.
- Navigation while manually locked is ignored.
- Navigation while a transition is active is ignored unless a future explicitly designed API changes that behavior.
- Navigation during cooldown is ignored.

Direction rules:

- Moving to a higher phase index sets direction to `"next"`.
- Moving to a lower phase index sets direction to `"prev"`.
- Idle and completed transitions use direction `"none"`.

Transition option resolution should be deterministic. If future implementation supports phase-specific transition settings, those settings should be resolved from the source phase at the moment navigation is accepted.

## Input handling expectations

Input hooks should translate browser input into flow navigation requests. They should not own scene behavior.

Expected input responsibilities:

- Attach browser event listeners only when enabled.
- Clean up listeners on unmount or option changes.
- Be safe to import in browser-oriented React apps and SSR-capable environments.
- Ignore input while the flow is locked.
- Ignore input while a transition is active.
- Respect cooldowns so repeated input does not cause accidental rapid navigation.
- Avoid hijacking typing in form controls or editable content for keyboard input.
- Support intentional `preventDefault` behavior where appropriate.
- Keep target handling explicit and testable.

Input hooks may expose convenience options, but future options should remain focused on browser input mapping and should not introduce animation or scene responsibilities.

## Transition locking behavior

There are two lock concepts that must remain distinct:

- Manual lock state: controlled by `lock()` and `unlock()` and reflected by `isLocked`.
- Transition guard state: navigation is ignored while `isTransitioning` is true.

Expected behavior:

- `isLocked` should represent manual locking, not merely the existence of an active transition.
- Manual locking prevents new navigation requests.
- Unlocking restores the ability to navigate only if no other guard blocks navigation.
- Active transitions should prevent overlapping transitions by default.
- Rejected navigation attempts should not reset progress, direction, cooldown, or phase state.

If future design introduces configurable transition locking, it must document how that option affects overlapping transitions, input hooks, React snapshots, and tests before implementation.

## Cooldown behavior

Cooldown protects against repeated accepted navigation caused by wheel bursts, repeated gestures, or rapid key presses.

Expected behavior:

- Cooldown starts only after navigation is accepted.
- Ignored navigation does not start, reset, or extend cooldown.
- Cooldown duration must be finite and non-negative.
- Navigation during cooldown is ignored.
- Cooldown should advance with the same deterministic update path used by the core flow machine.
- Cooldown may overlap with an active transition.
- If transition duration is longer than cooldown, transition locking still prevents overlap.
- If cooldown is longer than transition duration, navigation remains blocked until cooldown ends.

Input-level cooldowns, if present, should remain input throttling tools. Core transition cooldown remains the authoritative flow-level navigation guard.

## React responsibilities

React code is responsible for application-facing state and hooks.

`react/` should:

- Create and provide the flow machine through `FlowProvider`.
- Expose current flow state through context-backed hooks.
- Synchronize React snapshots after accepted imperative navigation calls.
- Synchronize React snapshots when frame-driven transitions complete.
- Throw clear errors when hooks are used outside `FlowProvider`.
- Avoid using React state as the per-frame animation store.
- Keep provider props stable and typed.

React components may use flow state for UI labels, buttons, panels, and conditional rendering. Frame-perfect scene animation should use the R3F bridge instead of relying on frequent React re-renders.

## R3F bridge responsibilities

`r3f/` connects the flow machine to React Three Fiber frame updates.

The R3F bridge should:

- Use R3F hooks only inside Canvas-bound components.
- Advance the core machine from `useFrame` with frame delta time.
- Provide frame callbacks with current flow progress and relevant flow state.
- Avoid storing every-frame progress in React state.
- Trigger React synchronization at meaningful lifecycle boundaries, such as transition completion.
- Remain rendering-agnostic: it should not prescribe meshes, cameras, shaders, materials, or effects.

Scene components should own visual interpretation. For example, a scene may map progress to object transforms, material uniforms, camera movement, or visibility, but those behaviors should not become built-in library effects.

## Public API boundaries

The public API should remain small and predictable. The current public API center is:

- `FlowProvider`
- `useFlow`
- `useFlowProgress`
- `useFlowFrame`

Input hooks may remain convenience APIs when they directly map browser input to flow controls, but v0.3.0 should avoid unnecessary public surface expansion.

This documentation task does not introduce any new public APIs. If future specs or PRs mention possible API changes, those changes must be clearly marked as future design considerations until implemented.

Any future public API change should include:

- rationale
- type shape
- runtime behavior
- migration impact
- README/docs updates
- tests
- whether the change is breaking

## TypeScript expectations

TypeScript should make phase-based flow safer without making the API difficult to use.

Expectations:

- Phase names should preserve literal union types when users pass const phase arrays.
- `goTo` should accept only known phase names where inference allows it.
- Internal core types should remain React-independent.
- Public types should be exported intentionally, not incidentally.
- Option types should reject invalid values through clear runtime validation where TypeScript cannot prove safety.
- Tests should cover type-sensitive behavior when practical.
- Breaking callback or option shape changes should be avoided unless a migration path is documented.

## Testing expectations

Future implementation PRs should include tests appropriate to the layer they change.

Core tests should cover:

- initialization
- invalid phase lists
- invalid initial phases
- `next`, `prev`, and `goTo`
- boundary no-ops
- unknown phase errors
- transition progress and completion
- direction changes
- manual lock and unlock
- transition guard behavior
- cooldown start, advancement, and ignored requests
- invalid timing options

React tests should cover:

- provider setup
- hook usage inside and outside provider
- snapshot updates after controls are called
- lock and unlock behavior surfaced through hooks
- progress reads through `useFlowProgress`
- provider recreation behavior when configuration changes

R3F bridge tests should cover:

- frame callback registration
- frame delta advancing the core machine
- callback payload shape
- transition completion synchronization
- hook errors outside `FlowProvider`

Input tests should cover:

- listener attachment and cleanup
- enabled/disabled behavior
- target handling
- threshold behavior
- direction mapping
- `preventDefault` behavior
- lock and transition guards
- cooldown guards
- keyboard repeat handling
- keyboard typing-target guards
- SSR/import safety where applicable

Because this PR is documentation-only, it should not require new runtime tests unless repository policy requires all checks to pass before merging.

## Implementation guidance for future PRs

Future v0.3.0 work should be split into small PRs with narrow ownership.

Recommended order:

1. Update or add core tests for lifecycle, locking, cooldown, and phase-specific timing expectations.
2. Implement core behavior in `core/` without React or R3F dependencies.
3. Wire provider props and hooks in `react/` only after core behavior is stable.
4. Update `r3f/` bridge behavior only for frame synchronization and callback state needs.
5. Update `input/` hooks separately for browser input ergonomics and guards.
6. Update README and examples only after behavior is implemented and tested.

Implementation rules:

- Keep source-layer boundaries intact.
- Avoid broad refactors unless they directly support the scoped behavior.
- Do not introduce visual effect abstractions.
- Do not add dependencies for behavior that can be implemented with small internal logic.
- Do not change package exports as part of documentation-only work.
- Do not change examples, tests, lockfiles, or runtime source files unless a future issue explicitly allows it.

## Acceptance baseline

For v0.3.0 planning, this document should be used as the baseline for review. A future implementation is aligned with this spec when it:

- keeps the project focused on phase/input/progress/state/frame updates
- preserves clear `core/`, `react/`, `r3f/`, and `input/` boundaries
- documents public API impact before changing exports
- includes tests for behavior changes
- avoids turning the library into an effects or animation framework
