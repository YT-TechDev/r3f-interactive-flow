# r3f-interactive-flow v0.3.0 Spec

## Status

Draft design baseline for v0.3.0. This document is intentionally documentation-only and does not change runtime behavior, package exports, examples, tests, dependencies, or the public API.

Target issue: `#81 - docs: add v0.3.0 spec`.

## Purpose

`r3f-interactive-flow` is a small React Three Fiber utility for building phase-based interactive 3D websites. It connects user input, scene phases, transition progress, React state, and React Three Fiber frame-based updates through predictable phase transitions.

The v0.3.0 baseline should help future implementation PRs preserve a clear separation of responsibilities:

- React manages application state, UI state, controls, and provider context.
- React Three Fiber manages frame-based visual updates inside Canvas-bound components.
- Core TypeScript logic manages deterministic phase transitions without depending on React, the DOM, or R3F.
- Browser input hooks translate user intent into flow navigation without owning scene behavior.

## v0.3.0 Goals

- Keep the library focused on predictable phase, input, transition, React, and R3F frame coordination.
- Keep the primary public API centered on `FlowProvider`, `useFlow`, `useFlowProgress`, and `useFlowFrame`.
- Preserve a small, predictable API surface and avoid introducing new public primitives without a dedicated design discussion.
- Define transition lifecycle behavior clearly enough that core, React, input, and R3F changes can be tested independently.
- Keep phase navigation deterministic for `next`, `prev`, and `goTo`.
- Define lock and cooldown semantics so input hooks and direct controls behave consistently.
- Clarify when React snapshots should update and when frame-driven values should stay outside React state.
- Provide implementation guidance for future PRs without implementing features in this documentation PR.

## Scope

v0.3.0 is in scope for refinements that directly support phase-based flow control:

- Core phase machine behavior.
- Transition duration, easing, direction, progress, and completion semantics.
- Transition locking and manual locking behavior.
- Cooldown behavior for accepted navigation.
- React provider, context, and hooks that expose stable flow state and controls.
- R3F bridge behavior for Canvas-bound frame updates.
- Browser input handling for wheel, touch, and keyboard navigation.
- TypeScript types that make phase names and public options safer.
- Tests for core logic, React hooks/provider behavior, R3F bridge behavior, and input hooks.
- Documentation that teaches the DOM UI to Canvas pattern.

## Non-Goals

v0.3.0 must not expand the package into a visual or animation framework. It should not add:

- visual effects collections
- `@react-three/drei` replacements
- particle systems or particle morph APIs
- camera preset systems
- shader effect libraries
- portfolio templates or full website templates
- visual editors
- general animation timelines
- GSAP wrappers
- Framer Motion wrappers
- router integrations
- Next.js-specific routing behavior
- heavy runtime dependencies
- unrelated example rewrites

Users remain responsible for their own scene content, camera behavior, materials, shaders, animation style, routes, and visual design.

## Expected Package Boundaries

The expected package structure remains:

```text
packages/r3f-interactive-flow/
  src/
    core/
    react/
    r3f/
    input/
```

### `core/`

`core/` contains React-independent TypeScript logic.

Responsibilities:

- Validate phase lists and initial phase values.
- Store the current phase index and transition state.
- Implement `next`, `prev`, `goTo`, `lock`, `unlock`, `update`, and `getSnapshot` behavior.
- Resolve transition timing, easing, and cooldown options.
- Keep behavior deterministic and unit-testable without React, DOM, R3F, or browser APIs.

Must not:

- Import React.
- Import React Three Fiber.
- Read browser events or DOM nodes.
- Manage Canvas lifecycle.

### `react/`

`react/` contains React Context, Provider, and hooks.

Responsibilities:

- Create and provide the core flow machine.
- Expose React-friendly snapshots and controls.
- Synchronize React snapshots after accepted imperative control calls and when frame updates complete transitions.
- Throw clear errors when hooks are used outside `FlowProvider`.
- Keep provider configuration stable and predictable.

Must not:

- Put frame-by-frame animation values into React state unnecessarily.
- Call R3F hooks such as `useFrame` or `useThree`.
- Attach browser input listeners directly unless the code belongs in `input/`.

### `r3f/`

`r3f/` contains React Three Fiber bridge hooks.

Responsibilities:

- Use R3F frame callbacks inside Canvas-bound components.
- Advance the core machine with R3F frame delta time.
- Provide current flow frame state to Canvas components for visual updates.
- Sync React snapshots only at meaningful lifecycle points, such as transition completion.

Must not:

- Call R3F hooks outside components rendered within `<Canvas>`.
- Attach DOM wheel, touch, or keyboard listeners.
- Own scene-specific animation, camera presets, shader effects, particles, or templates.
- Force frame-driven values through React state.

### `input/`

`input/` contains browser input handling.

Responsibilities:

- Translate wheel, touch, and keyboard events into `next` / `prev` navigation intent.
- Respect enabled/disabled options.
- Respect manual locks, active transitions, and cooldown expectations.
- Avoid hijacking normal browser controls, form fields, links, or explicitly ignored elements when such options are supported.
- Clean up event listeners reliably.
- Remain safe to import in non-browser environments.

Must not:

- Import React Three Fiber.
- Mutate 3D scene objects.
- Own transition progress or frame timing.

## Public API Boundary

The current public API should stay centered on:

- `FlowProvider`
- `useFlow`
- `useFlowProgress`
- `useFlowFrame`

Input hooks may remain convenience APIs for browser input integration, but v0.3.0 should avoid broad public API expansion.

This documentation PR does not introduce new public APIs. If future implementation work proposes new exports, renamed options, or breaking callback signatures, that work must be clearly marked as a future design consideration and handled in a dedicated implementation PR with tests and migration notes.

## Transition Lifecycle Behavior

A phase transition follows this lifecycle:

1. A navigation request is made through `next`, `prev`, or `goTo`.
2. The core machine validates whether the navigation can be accepted.
3. If accepted, the machine records the source phase, target phase index, direction, transition timing, easing, and cooldown values.
4. The public snapshot moves to the target phase immediately with `progress` reset to `0` and `isTransitioning` set to `true`.
5. Frame or time updates call `update(deltaMs)`.
6. Raw elapsed progress advances from `0` toward `1` and is clamped to `0..1`.
7. Easing is applied to in-progress values.
8. When raw progress reaches completion, `progress` is `1`, `isTransitioning` becomes `false`, and `direction` returns to `"none"`.
9. React snapshots are synchronized at control boundaries and transition completion, not on every frame unless a future PR explicitly designs that behavior.

Transition progress represents transition state, not an animation preset. Users decide how to map progress to mesh transforms, materials, camera movement, DOM UI, or other effects.

## Phase Transition Rules

- `next()` targets the next phase by index.
- `prev()` targets the previous phase by index.
- `goTo(phase)` targets an explicit phase from the configured phase list.
- Navigation outside phase bounds is ignored.
- Navigation to the current phase is ignored.
- Navigation while manually locked is ignored.
- Navigation while a transition is active is ignored unless a future explicit option changes that behavior.
- Navigation while cooldown remains active is ignored.
- Invalid `goTo` targets should throw a clear error rather than silently failing.
- Ignored navigation must not mutate phase, direction, progress, transition state, or cooldown state.
- Phase names should remain typed as string literal unions when users pass const phase tuples.

## Transition Locking Behavior

There are two distinct concepts:

- Manual locking through `lock()` / `unlock()`.
- Transition-time rejection of new navigation while `isTransitioning` is `true`.

Manual `isLocked` state should represent explicit user or application locking only. It should not become a catch-all flag for every reason navigation may be ignored.

Expected behavior:

- `lock()` prevents new navigation requests from being accepted.
- `unlock()` allows navigation again unless another guard still applies.
- Active transitions reject new navigation by default.
- Input hooks should check current flow state and avoid dispatching navigation while locked or transitioning.
- Future options such as configurable transition locking must be treated as future design considerations, not assumed v0.3.0 behavior unless implemented in a dedicated PR.

## Cooldown Behavior

Cooldown exists to prevent rapid repeated accepted navigation.

Expected behavior:

- Cooldown starts when navigation is accepted.
- Ignored navigation does not start, reset, or extend cooldown.
- Cooldown advances with elapsed update time.
- Cooldown may overlap with an active transition.
- A transition can complete before cooldown ends.
- New navigation remains ignored while cooldown remains active.
- Cooldown values must be finite numbers greater than or equal to `0`.
- Input hooks may have hook-local throttling or cooldown behavior only if it is explicitly designed, documented, and tested.

## Input Handling Expectations

Input hooks should translate browser events into navigation intent while preserving normal page behavior as much as possible.

General expectations:

- Default behavior should be safe for real pages with forms, buttons, links, overlays, and scrollable areas.
- Input hooks should support being disabled.
- Input hooks should clean up listeners on unmount or target changes.
- Input hooks should avoid dispatching navigation when flow is locked or transitioning.
- Input hooks should not contain R3F scene logic.
- Browser APIs must be guarded so package imports remain safe in SSR or non-browser test environments.

Wheel expectations:

- Wheel input maps intentional scroll direction to `next` or `prev`.
- Thresholds prevent accidental tiny deltas from navigating.
- Future target scoping, axis selection, ignored selectors, or hook-local cooldown changes should be separately designed and tested.

Touch expectations:

- Touch input maps intentional swipes to `next` or `prev`.
- Thresholds prevent accidental taps or tiny gestures from navigating.
- Touch cancellation and cleanup must be handled.
- Preventing default browser behavior should be conservative and documented.

Keyboard expectations:

- Keyboard input maps configured navigation keys to `next` or `prev`.
- Repeated keydown events should not cause repeated navigation unless explicitly designed otherwise.
- Typing in inputs, textareas, selects, or contenteditable regions should not unexpectedly navigate the flow.
- Preventing default browser behavior should only happen for handled navigation keys.

## React Responsibilities

React is responsible for:

- Owning provider configuration and user-facing flow controls.
- Rendering DOM UI that reflects the current phase and coarse transition state.
- Handling buttons, labels, accessibility text, panels, and non-frame-critical UI.
- Calling `next`, `prev`, `goTo`, `lock`, and `unlock` in response to application events.
- Reading `useFlowProgress` for UI that can update at React state cadence.

React is not responsible for:

- Updating mesh transforms every frame through React state.
- Running R3F frame loops.
- Owning scene-specific animation logic.

## R3F Bridge Responsibilities

The R3F bridge is responsible for connecting the flow machine to Canvas frame updates.

Expected behavior:

- `useFlowFrame` must run inside a Canvas-bound component.
- `useFlowFrame` advances transition time using R3F frame delta.
- The callback receives flow frame state and the R3F delta so users can mutate refs or other frame-local scene values.
- Frame-driven values should use refs or internal mutable state where appropriate.
- The bridge should not force frame progress through React state on every frame.

## TypeScript Expectations

- Phase lists should work well as const tuples.
- Hooks and controls should preserve `TPhase extends string` where practical.
- Public option types should be explicit and readable.
- Invalid configuration should fail early with clear errors.
- Internal types should stay close to the implementation and only be exported when they are useful for users.
- Type changes that could affect users should include tests or type-level assertions where practical.

## Testing Expectations

Future implementation PRs should include targeted tests for changed behavior.

Recommended coverage:

- Core tests for phase validation, duplicate phases, initial phase validation, `next`, `prev`, `goTo`, bounds, locks, cooldown, transition progress, easing, ignored navigation, invalid targets, and update delta validation.
- React tests for provider initialization, context errors, controls, snapshot synchronization, lock/unlock behavior, transition completion, and stable provider behavior.
- R3F bridge tests with mocked frame callbacks for update timing, callback payloads, transition completion sync, and hook errors outside the provider.
- Input tests for listener attachment, cleanup, thresholds, disabled state, default prevention, ignored targets, lock/transition guards, SSR-safe imports, and target handling.
- Public export tests when exports change.
- Typecheck and build validation for any TypeScript API changes.

Docs-only PRs should run lightweight available checks and clearly report any skipped commands or missing scripts.

## Implementation Guidance for Future PRs

- Keep implementation PRs small and scoped to one behavior area when possible.
- Update docs and tests alongside behavior changes.
- Do not mix DOM input logic into R3F scene logic.
- Do not call `useFrame`, `useThree`, or other R3F hooks outside Canvas-bound components.
- Do not use React state for values that change every frame.
- Prefer refs or internal mutable state for frame-driven visual values.
- Keep public APIs small, explicit, and predictable.
- Prefer explicit behavior over clever abstraction.
- Avoid runtime dependency additions unless absolutely necessary and separately justified.
- Preserve existing behavior unless a breaking change is intentionally designed, documented, and tested.
- Mark possible API changes as future design considerations until they are implemented.

None. This PR only adds or updates `docs/v0.3.0-spec.md` as a planning baseline. It does not change runtime source files, package exports, package metadata, lockfiles, examples, tests, dependencies, or implemented behavior.
